# PRD: Hero Carousel Refinement

**Date:** 2026-06-21
**Status:** Approved
**Related design:** `.docs/hero-design.md`

---

## Problem Statement

The homepage hero carousel does not match the reference site design in several ways: it spans full width above the sidebar instead of sitting side-by-side with it, it includes unnecessary left/right arrow buttons, it displays author information that clutters the overlay, and its dot indicators use a generic centered-circle style instead of the reference site's right-aligned dash style. The hero also uses a fixed pixel height that doesn't adapt proportionally to different viewport sizes.

## Solution

Restructure the homepage first-screen layout so the hero carousel and the sidebar (recent micro-post + popular tags) sit side-by-side on desktop, matching the reference site's two-column hero region. Simplify the carousel controls to dash-style dot indicators only (no arrows), remove author information from the slide overlay, and use a viewport-relative height for consistent visual impact across screen sizes.

## User Stories

1. As a visitor on desktop, I want to see the hero carousel and sidebar (micro-post + popular tags) side-by-side, so that the first screen feels dense and informative like the reference site.
2. As a visitor on desktop, I want the hero carousel to occupy roughly 70% of the first-screen width with the sidebar taking the remaining 30%, so that both elements have adequate space.
3. As a visitor, I want the hero carousel height to scale with my viewport (70vh), so that the visual impact is consistent regardless of my screen resolution.
4. As a visitor, I want to see only tag, title, and excerpt on each hero slide, so that the overlay is clean and readable without author clutter.
5. As a visitor, I want to manually switch hero slides by clicking dash-shaped indicators in the bottom-right corner, so that I can navigate without arrow buttons.
6. As a visitor, I want the hero to auto-advance every 5 seconds and pause on hover, so that I can browse featured content passively or at my own pace.
7. As a visitor, I want the dash indicator for the active slide to be visually distinct (brighter/longer), so that I always know which slide I'm viewing.
8. As a visitor, I want the hero to have rounded corners (12px) and a subtle border, so that it has clear visual separation from the page background.
9. As a visitor, I want the gradient overlay to fade from opaque at the bottom to transparent at the top, so that the cover image remains prominent while text stays readable.
10. As a visitor on mobile (<768px), I want the hero to be full-width at 70vh, so that the visual experience is immersive on small screens too.
11. As a visitor on mobile, I want the sidebar to appear below the article list (not directly below the hero), so that articles remain the primary scrolling content after the hero.
12. As a visitor, I want the hero section to be completely hidden when there are no featured posts, so that the page doesn't show an empty block.
13. As a visitor, I want the sidebar height to be driven by its content, not forced to match the hero height, so that there are no awkward empty gaps.

## Implementation Decisions

### Layout restructure

The current homepage template (`home.hbs`) renders the hero as a full-width block above a two-column grid (article list + sidebar). This changes to a two-column grid at the top level that places the hero and sidebar side-by-side, with the article list below in its own full-width row.

Desktop structure:
```
[Hero ~70%]  [Sidebar ~30%]
[Article list — full width ]
[Pagination                ]
```

Mobile structure (unchanged from current):
```
[Hero 100%]
[Article list]
[Sidebar]
```

This requires moving the sidebar partial invocation out of the current `.home-layout` grid and into a new wrapper that pairs it with the hero. The article list grid (`.post-feed`) remains its own section below.

### Hero markup changes

- Remove the `<button class="hero-arrow ...">` elements (both prev and next) from `hero.hbs`.
- Remove the `{{#primary_author}}...{{/primary_author}}` block from each slide.
- The `{{#get}}` call drops `include="authors"` since author data is no longer rendered; `include="tags"` is sufficient.
- Dot container markup stays but CSS restyling turns circles into dashes positioned bottom-right.

### Hero JS changes

- Remove all arrow-related event listeners and DOM queries from `hero.js`.
- Dot click handling, auto-advance (5s), hover pause/resume, and `render()` logic remain unchanged.

### Hero CSS changes

- Height: replace `height: 480px` with `height: 70vh` (desktop and mobile — the mobile-specific `height: 320px` override is removed).
- Arrow styles (`.hero-arrow`, `.hero-arrow-prev`, `.hero-arrow-next`, `.hero-arrow:hover`): delete entirely.
- Dot indicators (`.hero-dots`, `.hero-dot`, `.hero-dot.is-active`): restyle from centered circles to right-aligned horizontal dashes. The dots container moves from `left: 50%; transform: translateX(-50%)` to `right: 1.5rem; left: auto`. Each dot becomes a short horizontal bar (`width: 1.5rem; height: 3px; border-radius: 2px`) with the active dot brighter and wider (`width: 2.5rem`).

### New layout CSS

A new wrapper class (e.g. `.hero-row`) creates a `grid-template-columns: minmax(0, 1fr) 320px` grid pairing the hero and sidebar. On `<768px` this collapses to single column, and the sidebar uses CSS `order` to appear after the article list.

## Testing Decisions

This is a Ghost Handlebars theme with no unit test framework. Testing is visual and structural.

A good test verifies the externally visible rendering and behavior in a real browser — not internal template logic. Each test should confirm a single visual or interactive outcome.

### Visual verification (browser)

- Load a language homepage with at least one featured post; confirm the hero renders side-by-side with the sidebar on desktop viewport.
- Confirm hero height visually approximates 70% of viewport.
- Confirm no arrow buttons are visible anywhere in the hero.
- Confirm dot indicators appear in the bottom-right as horizontal dashes.
- Click a dot; confirm the carousel navigates to the corresponding slide.
- Wait 5+ seconds; confirm auto-advance occurs.
- Hover over hero; confirm auto-advance pauses.
- Resize to mobile viewport (<768px); confirm hero is full-width, sidebar appears below article cards.
- Load a homepage with zero featured posts; confirm the hero section is completely absent.

### Template structure verification

- Inspect rendered HTML: no `<button>` elements with `hero-arrow` class exist inside `[data-hero]`.
- Inspect rendered HTML: no author avatar or author name elements exist inside `[data-hero]`.
- Inspect rendered HTML: `hero-slide-tag`, `hero-slide-title`, `hero-slide-excerpt` elements are present in each slide.

## Out of Scope

- Right-side featured article list within the hero block (reference site has this; replaced by existing micro-post + popular tags sidebar).
- Changes to micro-post card dot indicators (they remain as circles).
- Sidebar content or styling changes (micro-post preview, popular tags — unchanged).
- Gradient overlay adjustments (kept as-is).
- Touch/swipe gesture support for mobile carousel navigation.
- Cross-language article mapping from the hero.

## Further Notes

- The hero currently does not render because no posts have the `featured` flag set in Ghost admin. To test, at least one post per language needs `Feature this post` enabled in the Ghost admin post settings panel.
- The sidebar height is content-driven and will not match the hero's 70vh. This is intentional — no filler or forced stretching is applied.
