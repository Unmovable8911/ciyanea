# PRD: Ciyanea — Multi-language + Micro-posts Ghost Theme

**Date:** 2026-06-14
**Source design:** `.docs/design.md`
**Ghost version:** 6.43.1
**Style reference:** https://enova.labs.enova.studio/ (style reference, not a pixel-perfect clone)
**Theme directory:** `ciyanea/` (currently empty — greenfield)

---

## Problem Statement

The author publishes content in seven languages (Chinese, English, German, French, Russian, Italian, Spanish) and writes two distinct kinds of posts: long-form articles and short Twitter-style "micro-posts." Out of the box, Ghost is a single-locale publication: it has no per-request language switching, no notion of a "micro-post" stream, and no theme that presents both content types side by side. The author wants an eye-catching, dark-themed site where:

- A visitor lands on the root and is automatically taken to content in their preferred language, with that preference remembered across visits.
- A visitor can switch language at any time from a single, always-visible control.
- Each language has its own homepage, article pages, tag pages, and a dedicated micro-post stream.
- The visual design feels designed and distinctive (elegant serif headings, clean body type, dark palette with a green accent), not like a generic default theme.

Today none of this exists; the theme directory is empty and `routes.yaml` is a stock single-collection file.

## Solution

A complete Ghost theme, **Ciyanea**, built from scratch inside `ciyanea/`, that:

- Uses **Ghost internal tags named by country/flag code** (`#cn`, `#us`, `#de`, `#fr`, `#ru`, `#it`, `#es`) plus a **URL-prefixed routes file** (`/cn/`, `/us/`, …) to partition all content by language, with no unprefixed default language. This single code per language drives the content filter, the URL prefix, and the flag image.
- Treats **micro-posts** as posts carrying both a language tag and the `#micro` internal tag, surfaced through a per-language `/<lang>/micro/` collection and the Ghost **Content API** (infinite scroll + a sidebar preview).
- Detects and remembers the visitor's language at the root path (`lang-redirect.hbs` + `lang-detect.js`) using `localStorage["preferred-lang"]` then `navigator.language`, falling back to English (`/us/`).
- Provides a **data-driven language switcher** sourced live from the language tags (`{{#get "tags"}}`), so adding a language never requires theme code changes — only a `routes.yaml` entry and a new tag in admin.
- Implements the full template set (homepage with hero carousel + article grid + sidebar, article detail with ToC + related posts, tag page, micro stream, static page, error pages) styled per the dark palette and typography in the design.
- Localizes all UI chrome via Ghost's `{{t}}` i18n and `locales/<lang>.json`.

## User Stories

### Language detection, switching, and routing

1. As a first-time visitor, I want the root URL `/` to send me to content in my browser's language, so that I see something I can read without choosing.
2. As a returning visitor, I want the site to remember the language I last chose, so that I skip language selection on every visit.
3. As a visitor whose browser language isn't supported, I want to be sent to the English site, so that I always land on readable content.
4. As a visitor, I want the root path to show no flash of content before redirecting, so that the transition feels instant and intentional.
5. As a visitor, I want a language control in the far right of the header on every page, so that I can change language at any time.
6. As a visitor, I want the language control to show the flag of the current language, so that I can tell at a glance which language I'm viewing.
7. As a visitor, I want clicking the language control to reveal all seven languages with their flag and native name, so that I can pick the one I want in its own script (e.g. "Русский", "Deutsch").
8. As a visitor, I want selecting a language to remember my choice and take me to that language's homepage, so that my preference persists and I start fresh in the new language.
9. As a visitor on a non-prefixed page (e.g. `/tag/...`), I want the switcher to reflect my stored preference, so that the control stays consistent; if there's no stored preference, no language is highlighted.
10. As the author, I want to add an eighth language later by only editing `routes.yaml` and adding a tag in admin, so that I never touch theme templates to grow language coverage.

### Homepage

11. As a visitor, I want a hero carousel of up to five featured posts in my language at the top of the homepage, so that the best content is showcased.
12. As a visitor, I want the hero to auto-advance every five seconds and let me move manually with arrows and dot indicators, so that I can browse featured content at my own pace.
13. As a visitor, I want the hero to disappear entirely when there are no featured posts, so that I never see an empty showcase.
14. As a visitor, I want a grid of article cards (excluding micro-posts) below the hero, so that I can browse recent long-form content.
15. As a visitor, I want each article card to show cover image, first public tag, title, a 4-line excerpt, author avatar + name, date, and reading time, so that I can decide what to read.
16. As a visitor, I want pagination showing "current / total" with older/newer links, so that I can navigate the full archive.
17. As a visitor, I want a sidebar with the most recent micro-post and a "view all micro-posts" link, so that I can sample short-form content from the homepage.
18. As a visitor, I want a popular-tags block in the sidebar, so that I can jump to topics that interest me.
19. As a visitor on a narrow screen, I want the sidebar to collapse below the main content, so that the homepage stays readable on mobile.

### Article detail

20. As a reader, I want the article page to show the title, cover image, and body, so that I can read the piece.
21. As a reader, I want a table of contents generated from the article's headings in the sidebar, so that I can jump to sections.
22. As a reader, I want a "featured posts" block (3, excluding the current post) in the sidebar, so that I can discover more highlighted content.
23. As a reader, I want an author bio below the article, so that I know who wrote it.
24. As a reader, I want up to three related articles in the same language (excluding the current one) at the bottom, so that I can keep reading.

### Tag page

25. As a visitor, I want a tag header with the tag name, description, and post count, so that I understand the topic and its size.
26. As a visitor, I want article cards and pagination on the tag page, so that I can browse all posts under that tag.
27. As a visitor, I want a popular-tags sidebar on the tag page, so that I can pivot to related topics.

### Micro-posts

28. As a visitor, I want a dedicated micro-post page per language at `/<lang>/micro/`, so that I can read all short-form posts in one stream.
29. As a visitor, I want the micro page to have a localized title at the top, so that I know I'm in the micro stream.
30. As a visitor, I want micro-posts in a single centered column (max-width ~640px), so that the stream reads like a focused feed.
31. As a visitor, I want each micro-post styled like a tweet (rounded border or left bar), with body truncated to 5 lines and click-to-expand, so that the stream stays scannable.
32. As a visitor, I want a micro-post's image (if any) shown full-width below its body, so that visual posts display properly.
33. As a visitor, I want a relative timestamp (e.g. "3 hours ago") in the bottom-right of each micro-post, so that I know how recent it is.
34. As a visitor, I want the micro stream to load more posts automatically as I scroll, so that I can browse without clicking pagination.
35. As a visitor, I want micro-posts to have no title and no profile link, so that they stay minimal and feed-like.
36. As a visitor on the homepage sidebar, I want the same expand/collapse behavior on the previewed micro-post, so that the interaction is consistent.

### Static and error pages

37. As a visitor, I want static pages (e.g. About) in a full-width centered single column with no sidebar, so that long-form content reads cleanly.
38. As a visitor hitting a missing URL, I want a localized 404 page with a return-to-homepage button, so that I can recover.
39. As a visitor hitting a generic error, I want a localized minimal error page with a return-to-homepage button, so that I'm not stranded.

### Visual design & localization

40. As a visitor, I want a consistent dark theme with a green accent driven by CSS custom properties, so that the site looks cohesive and can later support other themes.
41. As a visitor, I want elegant serif headings (Playfair Display) and a readable sans body (Inter), so that the site feels designed.
42. As a visitor, I want a responsive two-column → single-column layout at the defined breakpoints, so that the site works on any screen.
43. As a visitor, I want all UI chrome (nav labels, "reading time", "Table of Contents", "Related posts", pagination, error text, etc.) in my language, so that the interface matches the content language.

### Author / operability

44. As the author, I want the Content API key configured via theme custom settings (`{{@custom.content_api_key}}`), so that I can set it in admin without editing code.
45. As the author, I want each language tag to be named by its country/flag code (e.g. `#cn`, `#us`) and carry its native name (`description`), so that the switcher and detection are fully data-driven — the flag is loaded online from flagcdn using the code, with no image upload.

## Implementation Decisions

### Routing & language partitioning

- **Partition code:** each language is keyed by a single **country/flag code** stored as the internal tag's slug (`hash-<code>`); this one code drives the content filter, URL prefix, and flag. Codes: `cn, us, de, fr, ru, it, es` (Chinese → `#cn` → `/cn/`; English → `#us` → `/us/`).
- **Routes file** (`routes.yaml`, repo root — the file Ghost loads): root `/` renders the `lang-redirect` template. For each of the 7 languages, two collections:
  - `/<code>/` → `template: index`, `filter: "tag:hash-<code>+tag:-hash-micro"`, `permalink: /<code>/{slug}/`
  - `/<code>/micro/` → `template: micro`, `filter: "tag:hash-<code>+tag:hash-micro"`, `permalink: /<code>/micro/{slug}/`
  - Each collection declares `data: tag.hash-<code>` so the language's internal tag is bound into the template; `{{tag.slug}}` (value `hash-<code>`) is then used directly inside `{{#get}}` filters (e.g. `filter="featured:true+tag:{{tag.slug}}"`). This is the canonical server-side mechanism for "what language is this page" — `@request`/`@request.path` is **not** used (not a documented Ghost global).
  - `taxonomies`: `tag: /tag/{slug}/`, `author: /author/{slug}/`.
- **Filter syntax:** internal tag `#cn` is written `hash-cn`; `+` = AND, leading `-` = NOT.
- **No unprefixed default language.** Every language is reached only via its prefix.

### Templates & partials (built under `ciyanea/`)

- Top-level templates: `default.hbs`, `index.hbs`, `post.hbs`, `page.hbs`, `tag.hbs`, `error.hbs`, `error-404.hbs`, `micro.hbs`, `lang-redirect.hbs`.
- Partials: `post-card.hbs`, `micro-card.hbs`, `language-switcher.hbs`, `navigation.hbs`, `sidebar.hbs`, `hero.hbs`.
- `default.hbs` is the base layout (head, header with language switcher, footer) and globally loads `lang-switcher.js`.
- `sidebar.hbs` is context-aware: homepage/tag → recent micro-post preview + popular tags; article → ToC container + featured posts.
- `author.hbs` is intentionally absent (Ghost fallback handles author pages).

### Client-side JS responsibility split

- `lang-detect.js` — loaded only by `lang-redirect.hbs`; one-time detection (`localStorage` → `navigator.language` → `/us/` fallback) and redirect. The supported code set is injected by `lang-redirect.hbs` from a live `{{#get}}` of the language tags (not hard-coded in the JS).
- `lang-switcher.js` — global (via `default.hbs`); dropdown interaction, derives current code from URL prefix (`/cn/...` → `cn`), rewrites language-relative links declared as `data-lang-href="micro/"` into `/<code>/micro/`, writes `localStorage`, and redirects to `/<code>/` on selection.
- `hero.js` — loaded by `index.hbs`; 5-second auto-play, arrows, dot indicators.
- `toc.js` — loaded by `post.hbs`; extracts h2/h3 from the rendered article HTML and injects the ToC into the sidebar container.
- `micro-feed.js` — loaded by `micro.hbs`; Content API requests + infinite scroll; its expand/collapse logic is reused for the homepage sidebar micro-post preview.

### Language switcher data source

- Live query: `{{#get "tags" filter="visibility:internal+slug:-hash-micro" fields="slug,description"}}` — the language tags are the internal tags except `#micro`. Each option renders the flag `https://flagcdn.io/flags/1x1/<code>.svg` (code = slug minus `hash-`) + `description` (native name). Adding a language requires no theme edit.

### Content API

- Key surfaced via `{{@custom.content_api_key}}`, declared under `config.custom` in `ciyanea/package.json`. Read-only; safe client-side.
- Micro feed request shape:
  `GET /ghost/api/content/posts/?filter=tag:hash-<code>+tag:hash-micro&fields=html,published_at,feature_image&limit=20&page=<n>&key=<KEY>` (filter `+` URL-encoded as `%2B`). Sidebar preview uses the same query with `limit=1&page=1`.

### Hero & cards

- Hero data: `{{#get "posts" filter="featured:true+tag:{{tag.slug}}" limit="5"}}`; whole section wrapped in `{{#if}}` so it vanishes when empty.
- Article card: cover, **first public tag only** (internal/hash tags filtered out; tag row hidden if none), title, `line-clamp:4` excerpt, author avatar+name, date, reading time.
- Micro card: `line-clamp:5` body with JS expand, optional full-width `feature_image`, relative timestamp bottom-right, no title/profile link.
- Featured-posts sidebar block (article page): `{{#get "posts" filter="featured:true+id:-{{id}}" limit="3"}}`.

### Visual system

- CSS custom properties for the dark palette (`--color-bg #0f0f0f`, `--color-bg-card #1a1a1a`, `--color-text #f0f0f0`, `--color-text-muted #888`, `--color-accent #4ade80`, `--color-border #2a2a2a`) in `assets/css/screen.css`.
- Fonts: Playfair Display (headings) + Inter (body) via Google Fonts.
- Breakpoints: `>1024px` two-column; `768–1024px` two-column with narrower sidebar; `<768px` single column with sidebar below.

### Localization

- Ghost `{{t}}` i18n with `locales/<lang>.json` for all 7 languages. Note: Ghost i18n is site-wide (single publication locale); multilingual *content* is driven by routes + internal tags, while `{{t}}` covers UI chrome strings only. Locale files are named by **language** code (`zh, en, …`), distinct from the route/flag code (`cn, us, …`) — they differ only for Chinese and English.

### Pagination

- Custom markup using `{{pagination}}` vars: `{{page}} / {{pages}}`, `{{#if next}}`/`{{#if prev}}` links, labels via `{{t}}`.

## Testing Decisions

**Chosen seam (confirmed with author): rendered-HTML assertions against the running local Ghost instance** at `http://localhost:53368`. Tests issue HTTP requests to real theme-rendered routes and assert on the returned DOM structure. This is the highest meaningful seam for a Ghost theme — it exercises `routes.yaml`, the template/partial composition, and the `{{#get}}`/filter wiring end-to-end without coupling to Handlebars internals.

- **What makes a good test here:** assert on *externally observable rendered output*, not template internals — e.g. "the `/cn/` homepage response contains a hero region only when featured posts exist," "article cards expose title/excerpt/author/date/reading-time elements," "the language switcher renders 7 options each with a flag image and native-name text," "`/<code>/micro/` renders the localized page title and a centered micro-stream container," "a missing URL returns the localized 404 with a return-home link," "pagination renders `page / pages` plus older/newer links when applicable." Do not assert on class names that are pure styling hooks or on exact whitespace.
- **Modules covered by this seam:** all server-rendered templates and partials — `index`, `post`, `tag`, `micro`, `page`, `error`/`error-404`, `lang-redirect` (assert it renders the no-content shell + the detection script tag), and the language-switcher partial (assert the 7 data-driven options).
- **Routing coverage:** assert each language prefix resolves (`/<code>/` and `/<code>/micro/`) and that the index collection excludes micro-posts while the micro collection includes only them.
- **Prior art:** none in this repo yet (greenfield). Establish the first rendered-HTML test harness; keep it dependency-light (HTTP fetch + an HTML/DOM assertion helper) and pointed at the docker-served URL from `CLAUDE.md`.

**Explicitly NOT covered by the chosen seam (verified manually against `.screenshots` and the reference site):** purely client-side JS behaviors that produce no distinguishing server-rendered HTML — root-path language detection/redirect outcome, language-switcher dropdown interaction + `localStorage` write, hero auto-play/manual switching, micro infinite-scroll loading, micro expand/collapse, and `toc.js` ToC generation. These are confirmed by manual visual checking; automated browser testing was not selected for this PRD.

## Out of Scope

- Cross-language article linking (the switcher always lands on the language homepage; no per-article translation mapping).
- Micro-post individual detail pages (Ghost URLs exist but are never linked from the theme).
- Ghost Members / subscriptions.
- Comments.
- A custom `author.hbs` (Ghost fallback is used).
- Micro-post authoring UI (done in Ghost admin).
- Automated browser/JS behavior testing and gscan validation (not the selected seam; gscan may still be run ad hoc but is not a PRD deliverable).
- Light theme / alternate palettes (CSS variables leave room for it, but only the dark theme ships).

## Further Notes

- The theme directory `ciyanea/` is empty; `routes.yaml` is currently the stock single-collection file and must be rewritten to the prefixed multi-language form. The theme is already symlinked into Ghost, so changes are picked up by the running docker instance.
- Each language's internal tag must be configured in Ghost admin named by its country/flag code (`#cn`, `#us`, `#de`, `#fr`, `#ru`, `#it`, `#es`) with the native name as `description`. No image is uploaded — flags are loaded online from `https://flagcdn.io/flags/1x1/<code>.svg`. The switcher and root detection depend on these tags existing.
- Reply to the author in Simplified Chinese; all files and code are written in English (per `CLAUDE.md`).
- Adding a language later = add the `routes.yaml` collection pair + create the `#<code>` internal tag in admin + add `locales/<lang>.json`; no template changes.
