# PRD: Ciyanea — Header Navigation Bar + Search

**Date:** 2026-06-15
**Ghost version:** 6.43.1
**Style reference:** `.screenshots/home-top.png` (Inova/Enova header — style reference, not a pixel-perfect clone)
**Theme directory:** `ciyanea/`
**Current-state evidence:** `home.png` (theme as built today — header has logo + empty nav + language switcher only)
**Builds on:** `.docs/prd/2026-06-14-13:31.md` (the multi-language + micro-post theme)

---

## Problem Statement

The theme's top header today shows only the site logo on the left and the language switcher (flag) on the far right. The middle `<nav>` is an empty placeholder ("filled in a later slice"), and there is no search anywhere on the site. Compared with the reference site, the header feels unfinished: a visitor who lands on a language homepage has no in-page way to move between the two content streams the theme is built around (long-form articles and the micro-post stream), no way to reach static pages, and no way to search the publication. The author wants the header to look and behave like a real navigation bar — menu links plus a search affordance — while respecting the theme's existing per-language URL architecture.

## Solution

Fill in the header so that, on every page, the visitor sees:

- A **navigation menu** immediately to the right of the logo, containing two **language-aware** built-in items — **Home** (`/<code>/`) and **Micro** (`/<code>/micro/`) — followed by any **global page links** the author configures in Ghost admin (e.g. About), which are left un-prefixed because pages and tags are not partitioned by language.
- A **search control** (magnifying-glass icon button) in the right-hand group, left of the language switcher, that opens Ghost's built-in search (sodo-search) overlay.
- The language switcher remains where it is, at the far right.

The header is **sticky** (stays pinned on scroll). The currently-active section (Home or Micro) is **highlighted**. On narrow screens the menu collapses into a **hamburger** toggle, while the search icon and language switcher stay visible.

The built-in Home/Micro links reuse the theme's existing `data-lang-href` mechanism (already implemented in `lang-switcher.js`), so they automatically resolve to the active language's prefix. The built-in labels are localized through Ghost's `{{t}}` i18n across all seven locales.

## User Stories

### Navigation — language-aware built-in items

1. As a visitor on any language page, I want a navigation menu next to the logo, so that I can move around the site without scrolling back to find links.
2. As a visitor, I want a **Home** link that takes me to the homepage of the language I'm currently viewing, so that I stay within my chosen language.
3. As a visitor, I want a **Micro** link that takes me to the micro-post stream of the language I'm currently viewing, so that I can switch between long-form and short-form content in my language.
4. As a visitor who switches language, I want the Home and Micro links to point at the new language's pages afterwards, so that the menu always matches the language I'm reading.
5. As a visitor, I want the navigation label text to appear in my current language (e.g. "Startseite" rather than "Home" on the German site), so that the chrome feels native.
6. As a visitor, I want the menu item for the section I'm currently in (Home or Micro) to be visually highlighted, so that I know where I am.
7. As a visitor on the root path `/` (the redirect shell) or a non-prefixed page, I want the Home link to still resolve to a sensible destination, so that I am never given a broken link before a language is determined.

### Navigation — author-managed global pages

8. As the author, I want links I add to Ghost's primary navigation (e.g. About, Contact) to appear in the header after Home and Micro, so that I can manage page links from admin without editing the theme.
9. As a visitor, I want page links (About, etc.) to take me to the single global page, so that I reach the content directly and never hit a 404 from a language prefix being added to a page that has no per-language route.
10. As a visitor, I want tag links or external links placed in admin navigation to behave normally (no language prefix), so that they resolve correctly.
11. As the author, I want the header to render correctly even when I have configured zero primary-navigation links, so that the theme never shows an empty or broken menu region.

### Search

12. As a visitor, I want a search icon in the header on every page, so that I can find content from anywhere on the site.
13. As a visitor, I want clicking the search icon to open Ghost's search overlay, so that I can type a query and jump to matching posts, pages, tags, or authors.
14. As a visitor, I want the search affordance to be reachable by keyboard and labelled for assistive technology, so that search is accessible.
15. As a visitor, I accept that search results span all languages (not filtered to my current language), so that I can still find anything published on the site. *(Cross-language results are an accepted limitation of using Ghost's built-in search.)*

### Layout, sticky behavior, and responsiveness

16. As a visitor, I want the logo and navigation grouped on the left and the search and language switcher grouped on the right, so that the header reads cleanly like the reference design.
17. As a visitor, I want the header to stay pinned to the top of the viewport as I scroll, so that navigation and search are always within reach.
18. As a visitor on a narrow screen, I want the navigation menu to collapse into a hamburger button, so that the header stays uncluttered on mobile.
19. As a visitor on a narrow screen, I want to tap the hamburger to reveal the menu and tap again (or tap outside / press Escape) to dismiss it, so that I can navigate on mobile.
20. As a visitor on a narrow screen, I want the search icon and language switcher to remain visible alongside the hamburger, so that I keep access to search and language without opening the menu.
21. As a visitor, I want navigation links to show a clear hover state (matching the theme's green accent), so that interactive elements feel responsive.

### Internationalization

22. As the author, I want the built-in Home and Micro labels and the search control's accessible label to be translated in all seven locales, so that the header is fully localized.
23. As the author, I want to add an eighth language later without changing header markup, so that the header keeps working as the theme grows (the built-in items derive the prefix at runtime; only a new locale file and routes entry are needed for the new label).

## Implementation Decisions

- **Header is restructured into two flex groups.** The header inner becomes a left group (logo + navigation) and a right group (search button + language switcher), replacing the current three-item `space-between` arrangement. The language switcher stays rightmost.

- **Hybrid navigation source.** The two language-aware items (Home, Micro) are rendered by the theme itself, not pulled from Ghost admin. The author-managed page links come from Ghost's primary navigation helper, appended after the built-in items. This avoids the otherwise-impossible task of reliably distinguishing, from a flat list of admin URLs, which links must be language-prefixed and which must not.

- **A dedicated navigation partial** holds the built-in items plus the admin-navigation loop, and is included by the main layout in place of the current empty `<nav>` placeholder.

- **Built-in links reuse `data-lang-href`.** Home is marked so it resolves to `/<code>/` and Micro to `/<code>/micro/`, using the existing client-side rewrite already implemented in `lang-switcher.js` (`data-lang-href="…"` → `/<active>/…`). No new prefixing mechanism is introduced. Home uses an empty relative value so it resolves to the bare language root.

- **Static fallback hrefs.** Because the `data-lang-href` rewrite only fires once an active language code is known (from the URL prefix or stored preference), the built-in items carry a sensible static `href` fallback (Home → site root `/`) for the pre-rewrite state and for the root redirect shell.

- **Page/tag/external links are passed through unchanged.** The admin-navigation loop renders ordinary anchors with the URLs Ghost resolves; no language prefix is applied to them. Pages and tags remain global (not partitioned by language) in this iteration.

- **Search uses Ghost's built-in sodo-search.** A button carrying the `data-ghost-search` attribute triggers Ghost's search overlay; the theme does not build a custom search UI or query the Content API for search. Search results are site-wide and not language-scoped. (Implementation must confirm the sodo-search bundle loads on the page; if it is not auto-injected, the theme will opt in via the supported mechanism.)

- **Sticky header.** The header uses CSS sticky positioning so it remains visible while scrolling.

- **Active-section highlight.** The navigation marks Home or Micro as current based on the current path's language-relative segment (root of the language → Home; `…/micro/…` → Micro). This determination reuses the same current-language/path parsing already present for the switcher rather than introducing a separate scheme.

- **Mobile hamburger.** Below the existing 768px breakpoint, the navigation list collapses behind a toggle button; the search button and language switcher remain in the bar. Opening/closing follows the same interaction conventions already used by the language switcher dropdown (toggle click, outside-click to close, Escape to close).

- **Client-side behavior placement.** The hamburger toggle and active-section highlight are implemented in small vanilla JS consistent with the existing per-feature script files; the language-prefix rewrite continues to be owned by `lang-switcher.js`. Path/prefix/active-state logic is factored so it can be exercised as pure functions (see Testing Decisions).

- **Localization keys.** New i18n keys for the built-in labels (Home, Micro) and the search control's accessible label are added to all seven locale files (en, zh, de, fr, ru, it, es), following the existing `{{t}}` usage already present for "Main navigation".

## Testing Decisions

**What makes a good test here:** assert externally-observable behavior of the header — what links resolve to, what is visible/highlighted, and what user actions (click search, toggle hamburger) produce — not internal function names, DOM structure incidental to styling, or CSS property values. Tests should be resilient to markup refactors that preserve behavior.

**Primary seam — Playwright E2E against the local Ghost instance (`http://localhost:53368`):**
- The built-in **Home** and **Micro** links resolve to the correct language-prefixed URLs on at least two different language collections (e.g. `/us/…` and `/cn/…`), confirming the prefix tracks the page's language.
- An admin-configured **page link** is rendered **without** a language prefix.
- The **search button** is present, is keyboard-focusable, carries its accessible label, and clicking it opens Ghost's search overlay.
- The **current section** (Home vs Micro) is highlighted correctly when viewing a language homepage versus a micro-stream page.
- At a **mobile viewport**, the navigation is collapsed behind the hamburger, the search and language controls remain visible, and toggling the hamburger reveals/dismisses the menu.
- The header remains **pinned** (sticky) after scrolling.
- The header renders without error when **no admin navigation** is configured.

**Secondary seam — `node --test` unit tests (the project's declared `test/*.test.mjs` script, currently with no tests yet):**
- Pure functions for: deriving the active language code from a path/stored preference; building a language-prefixed href from a `data-lang-href` value + active code; and determining whether a given path marks Home or Micro as the active section.
- These functions must be factored to be importable/testable without a DOM, mirroring how the existing client logic parses the URL.

**Prior art:** there are no existing automated tests in the repo yet; the `node --test` script in `package.json` establishes the intended unit-test runner, and the project already has browser-driven testing capability (Playwright via the webapp-testing toolkit) for the E2E seam. This feature establishes the first tests at both seams.

## Out of Scope

- **Per-language filtering of tag pages and static pages.** Tag pages remain global `/tag/{slug}/`; pages remain single global documents. Making these language-aware (via enumerated per-language collections or client-side Content-API filtering) is a separate, larger effort with significant SEO/architecture tradeoffs and is explicitly deferred.
- **Custom, language-scoped search.** Building a Content-API-backed search that filters to the current language (using the already-wired `content_api_key`) is not done; Ghost's built-in cross-language search is used instead.
- **Membership/account controls.** No bookmark/saved-posts icon, no "Sign in", no "Join now" button — even though the reference shows them. The header stays content-only.
- **Secondary navigation** (Ghost's second nav menu) and footer navigation.
- **Pixel-perfect cloning** of the reference header; it is a style reference only.
- **Changes to routes.yaml**, the language detection/redirect flow, or the language switcher's own behavior beyond what the built-in nav links need.

## Further Notes

- The `<body>` already carries `data-content-api-key`, originally wired for the micro-post feed. It is intentionally **not** used by this feature (search is delegated to Ghost), but it remains available should a future language-scoped search be built.
- The reference header's many menu items (Home, Pages, Homepage Layouts, Navigation Style, Membership, More…) are demo content from the reference theme and carry no requirements for Ciyanea; only the *shape* of the header (logo + menu on the left, search on the right) is borrowed.
- Existing breakpoints in the stylesheet are 1024px and 768px; the mobile collapse aligns to the 768px breakpoint already in use.
- The empty `<nav class="site-nav">` placeholder and its existing `"Main navigation"` localized `aria-label` are the insertion point; the new partial replaces the placeholder's contents.
